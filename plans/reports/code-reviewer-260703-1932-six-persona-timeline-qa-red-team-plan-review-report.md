# Red-team review — six-persona timeline QA plan

Plan: `plans/260703-1930-six-persona-timeline-qa/plan.md`

## Findings

| # | Severity | Finding |
|---|---|---|
| 1 | **Blocking** | Step 5 grounding is wrong. Plan cites `work-shift-attendance.spec.ts` for "teacher marks (student) attendance." Read that spec (`apps/e2e/tests/work-shift-attendance.spec.ts`): it tests staff **chấm công** (clock-in/out via WiFi IP, shift categories) — a completely different feature from marking student attendance in a class session (`attendance-panel.tsx`, `attendance-roster.tsx`, `apps/api/src/routers/attendance.ts`). No existing e2e spec touches student-session attendance marking at all. Step 5 is NOT "already individually tested" as the plan's core premise claims — it's a net-new, unvalidated UI path. |
| 2 | **Blocking** | Step 4→5 wiring gap the plan didn't fully surface. Read `apps/api/src/routers/finance.ts` `receiptApprove` (~line 851): enrollment is created only `if (receipt.classBatchId)` — comment explicitly warns "Enrolling a student into an unrelated batch corrupts attendance records." But `receiptCreate`'s `classBatchId` is `z.string().uuid().optional()`, and neither grounding spec (`admin-commission-chain.spec.ts`, `admin-receipt-provision.spec.ts`) ever sets it — `admin-commission-chain.spec.ts` only fills "Khóa học" (course), not a class batch. So chaining Steps 2→4→5 as scripted will provision a student with **no enrollment**, and Step 5 (attendance) has nothing to mark against. The plan's seed-audit table gestures at "new student ≠ TEST-001" but never states that the draft-receipt dialog must additionally select a class batch, nor confirms the admin UI dialog even exposes that field in the "from opportunity" flow used by the grounding spec. |
| 3 | **Blocking** | CI/CD soak-wait math (Part A) is wrong by ~24h. Verified `plans/260703-0022-devops-tier1-hardening/plan.md`: `status: soaking`, `soakStartedAt: 2026-07-02T19:58:30Z`, `soakRequiredHours: 48` → deadline is correctly stated as `~2026-07-04T19:58:30Z`. But from the plan's own reference point (`2026-07-03T12:25 UTC`) to that deadline is **~31.5 hours**, not "~7.5h remaining" as written. The "no action needed, wait ~7.5h" framing understates the wait by a full day — low risk to safety (still correctly says "no action needed"), but the plan will mislead whoever re-checks based on the stated ETA. |
| 4 | **Should-fix** | Dependencies section omits a migration step. Root `package.json` has a distinct `db:migrate` script (`pnpm --filter @cmc/db migrate`) separate from `db:seed`. The plan's Dependencies list only says `docker compose ... up -d` then `pnpm db:seed`. If `cmcnew_pgdata` is a fresh/empty volume (plausible — plan itself notes the stack is "currently NOT running"), seed will fail with missing-table errors until migrations run first. `packages/db/.env`'s `DATABASE_URL` uses role `cmc_app`, which is created inside a migration (`20260623045316_rls_tenancy/migration.sql`), not by `docker-compose.dev.yml`'s `POSTGRES_USER`/`PASSWORD` (`cmc`/`cmc`) — confirming migrate must run before seed/RLS-scoped queries work, not merely "start the container." |
| 5 | **Minor** | DB isolation claim (verify item #3) is moot rather than wrong: neither `admin-commission-chain.spec.ts` nor other checked specs perform any `beforeAll`/reset/truncate — they rely on `Date.now()`-suffixed unique names to avoid collision on a persistent shared dev DB, and `playwright.config.ts` runs `fullyParallel: false` / 1 worker. So there's no "independent per-spec reset" to conflict with chaining — but this also means the dev DB accumulates data run-over-run indefinitely; the plan should note this as an ongoing pollution risk for the new chained spec, not just a one-time concern. |
| 6 | **Confirmed correct** | Part A's core answer (Jenkinsfile only deploys `main`, `develop` has no deploy stage, blocked-by relationship, file-collision reasoning) checks out structurally against `plans/260703-0022-devops-tier1-hardening/plan.md`'s actual `blockedBy`/status fields. Only the elapsed-time arithmetic (#3 above) is wrong. |
| 7 | **Confirmed correct** | `receiptApprove`'s O4→O5 auto-win and LMS `StudentAccount` provisioning (Steps 3-4) are real and in the claimed order — verified directly in `finance.ts`: atomic claim of draft → student provisioning → conditional enrollment → LMS account creation → opportunity stage flip to `O5_ENROLLED` inside one transaction. |

## Unresolved Questions the plan should add

1. Given finding #1, is Step 5 in scope for this round as "just run an already-tested step," or does it need to be reclassified as **new, unvalidated UI surface** (higher scrutiny, possibly its own red-team pass before being chained)?
2. Given finding #2, does the "Tạo phiếu thu" dialog (draft-receipt-from-opportunity) even expose a class-batch picker in the current UI? If not, Step 5 cannot be exercised via the Step 1-4 chain as scripted and needs either a UI change or a manual enrollment step inserted between Step 4 and Step 5 (which the plan currently forbids by design ("chained... not independently seeded per step")).
3. Add `pnpm db:migrate` to the Dependencies section before `pnpm db:seed`.
4. Correct the Part A wait estimate to ~31.5h (deadline `2026-07-04T19:58:30Z` from a `2026-07-03T12:25 UTC` reference point), not ~7.5h.
