## Human checkpoints (non-automatable, autonomous execution must pause here)

Found during a pre-handoff soundness review (2026-07-03): several of this phase's steps require a
human to act outside this repo/session — no API credential for these exists in the codebase
(`grep` confirms zero `CF_API_TOKEN`/Cloudflare API wiring anywhere). An autonomous agent cannot
complete these; it must pause, clearly state what's needed, and wait for the operator:

1. **Entra redirect URI registration** (step 3) — adding the dev callback URL to the Azure AD app
   registration requires Entra/Azure admin portal access. Not scriptable from this repo.
2. **Cloudflare SSL/TLS mode + Origin Certificate confirmation** (steps 4-5) — must be read from the
   Cloudflare dashboard (or a scoped `CF_API_TOKEN` provisioned first, which doesn't exist today).
3. **Entra client secret for the dev app registration** (feeds Phase 2's dev `.env`) — only
   obtainable from Azure AD by a human; generic session secrets are agent-generatable, this one isn't.
4. **Real-browser SSO/cookie inspection** (Phase 5) — logging in through Entra interactively (likely
   MFA-gated) with a real staff account cannot be automated by the agent.

Everything else in this plan (DNS lookups, compose file authoring, Jenkinsfile edits, health-check
verification) IS agent-executable without a human. When an autonomous session reaches one of the 4
items above, it should stop that specific step, log exactly what's needed and why, and continue with
any other non-blocked work rather than treating the whole run as failed.

## Requirements

- Functional: confirm dev hostnames, SSO redirect ownership, Cloudflare SSL/TLS mode, and current prod health baseline.
- Non-functional: do not expose secrets in repo; do not change prod routing during this phase.
- Governance: create a durable decision record before implementing live architecture changes.

## Architecture

The implementation should keep these boundaries:

```text
Cloudflare DNS
  -> VPS nginx edge
      -> prod app services for erp/hoc
      -> dev app services for deverp/devlms

Entra ID
  -> prod redirect URI for prod ERP
  -> dev redirect URI for dev ERP
```

The Entra application may be shared if the app supports multiple redirect URIs,
but secrets, cookie names, app origins, and database URLs must be separate.

## Risk Assessment

- Risk: wrong ERP dev domain leads to a valid plan for the wrong hostname.
  Mitigation: stop if `deverp.edu.vn` is required but still unresolved.
- Risk: SSO redirects to prod from dev.
  Mitigation: do not enable dev SSO until redirect URI and app origin envs are separate.
- Risk: TLS strategy drift between this plan and 260703-0022's already-locked decision.
  Mitigation: this plan uses Cloudflare "Full" + self-signed origin cert (0029), matching 0022 —
  do not switch to "Full Strict" independently; extend `ensure-origin-cert.sh`'s SAN list for the new
  dev hostnames rather than adopting a different cert strategy.
