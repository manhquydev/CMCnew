# Red-Team Review — DevOps Tier-1 Hardening Plan (Assumption Destroyer)

Reviewer: code-reviewer | Plan: `plans/260703-0022-devops-tier1-hardening/` | Verification tier: Standard (Fact Checker + Contract Verifier)

All claims below were checked against the live repo (grep/read) and, where the plan's premise was empirically checkable, against a real `docker compose` (v5.1.4 CLI) test run in the scratch sandbox. Verification status is marked inline.

## Finding 1: Memory budget ignores Docker image-build overhead during every main-branch deploy

- **Severity:** Critical
- **Location:** Phase 2, section "Sizing arithmetic" / "Proposed limits"
- **Flaw:** The budget (`plans/260703-0022-devops-tier1-hardening/phase-02-resource-limits.md:16-45`) sums only the **steady-state container ceilings** (postgres 1g + api 1g + redis 256m + nginx 128m + admin 128m + lms 128m = 2.64 GiB) against "OS + docker daemon ~0.8 GiB" + "Jenkins 3.0 GiB" = 6.4 GiB worst case, 1.4 GiB headroom. It never accounts for the memory consumed by the **`docker build`/BuildKit processes themselves** that run on every main deploy.
- **Failure scenario:** `Jenkinsfile:58` runs `$COMPOSE up -d --build` on every merge to `main`. This triggers `docker build` for `api`, `admin`, and `lms` concurrently (Compose v2 parallelizes builds by default). VERIFIED(`apps/api/Dockerfile:37-42`): full-workspace `pnpm install --frozen-lockfile` + `pnpm --filter @cmc/db run generate`; VERIFIED(`apps/admin/Dockerfile:42`, grep): `pnpm --filter @cmc/admin run build` (a Vite/esbuild bundle — notoriously memory-hungry, commonly 500 MB–1.5 GB peak per process). These BuildKit build processes are **not containers governed by `deploy.resources`** — they are unbounded host processes that run *while the OLD `api`/`admin`/`lms`/`postgres`/`redis`/`nginx` containers are still live and serving traffic* (compose only stops/recreates a service after its new image finishes building). So the real worst-case is: old steady-state stack (~2.64 GiB) + Jenkins (up to 3 GiB) + OS (~0.8 GiB) + **3 concurrent unbounded builds** — easily exceeding the claimed 1.4 GiB headroom and pushing the box past 7.8 GiB with no swap, triggering a **host-level OOMKiller** (which can kill *any* process, including Postgres or the Jenkins JVM, not just the offending build) rather than the intended, contained container-level OOMKill.
- **Evidence:** `plans/260703-0022-devops-tier1-hardening/phase-02-resource-limits.md:20-27` (budget arithmetic); `Jenkinsfile:43-58` (`up -d --build` on every main merge, no `--build`-scoped resource cap); `apps/api/Dockerfile:37-42`; `apps/admin/Dockerfile:42`. Corroborated by the plan's own cited source, `plans/reports/brainstorm-260703-0044-dev-domains-vps-capacity-deploy-policy-report.md:47`: *"Not enough to casually run heavy E2E, parallel builds, and prod traffic together without caps."* — the plan cites this report as a source but doesn't fold its own build-memory warning into the Phase 2 budget.
- **Suggested fix:** Either (a) budget an explicit build-memory reserve (e.g. treat concurrent builds as a 4th "Jenkins-adjacent" bucket, not zero), or (b) serialize builds (`docker compose build --parallel 1` equivalent, or build before stopping old containers is unavoidable — but at minimum stop parallelism), or (c) add the build-time window to the Phase 4 soak explicitly (currently the soak in Phase 4 Step 4 only watches steady-state `docker stats`, not the deploy/build window itself).

## Finding 2: Phase 4 Step 1 vs Step 3 give two contradictory merge paths for the same Jenkinsfile line, risking a broken main deploy

- **Severity:** High
- **Location:** Phase 4, sections "Step 1" and "Step 3"; Phase 3 "Jenkinsfile joint-ownership" note
- **Flaw:** Phase 4 Step 1 says: *"Merge the Jenkinsfile `publishChecks` change (Phase 3 §1) — **jointly with Phase 1's Jenkinsfile edit** if on one branch."* (`phase-04-rollout-validation.md:19`). But Phase 1's Jenkinsfile edit is exactly the line that replaces `docker volume create cmcnew-prod_letsencrypt` with `bash scripts/ensure-origin-cert.sh` (`phase-01-tls-reconciliation.md:42`) — and `scripts/ensure-origin-cert.sh` is a **new file** whose creation is not scheduled until Step 3 (*"Merge Phase 1 (helper script + Jenkinsfile deploy step + certbot profile + docs/decision)"*, `phase-04-rollout-validation.md:30`). If an operator takes the "jointly" branch offered in Step 1, the Jenkinsfile on `main` will reference a script that doesn't exist yet in the repo at that point in the sequence.
- **Failure scenario:** Operator merges the joint branch at Step 1 (as explicitly permitted by the plan's own wording). The next `main` build's `Build + Deploy` stage runs `bash scripts/ensure-origin-cert.sh` → file not found → the deploy stage fails → **every subsequent main-branch deploy is broken** until Step 3's script actually lands, and — worse — the just-wired `CMCnew CI` check (Step 1's own stated goal, "confirm it reflects the build result") now reports FAILURE for a reason ("Build+Deploy" only runs on `main`, not on PRs, so this specific failure wouldn't hit PR checks — but it does hit the live deploy pipeline used to validate Step 1 itself, and blocks the "next real main deploy" Step 3.3 depends on).
- **Evidence:** `phase-04-rollout-validation.md:19` ("jointly with Phase 1's Jenkinsfile edit if on one branch") directly conflicts with `phase-04-rollout-validation.md:29-33` (Step 3 owns landing Phase 1, including "helper script"). `phase-03-ci-merge-gate.md:28` only says "Author both on one branch or land Phase 1 → Phase 3" — it does not resolve which of Phase 4's two steps that joint branch is supposed to land in.
- **Suggested fix:** Pick one explicit path in the plan text: either (a) Step 1 merges *only* the `publishChecks` hunk of the Jenkinsfile (surgically split from Phase 1's deploy-stage hunk), or (b) drop the "jointly" option entirely and hard-serialize Phase 3 → Phase 1 → Step 3, contradicting nothing. Currently both are simultaneously "allowed" by the plan text, which is the actual defect.

## Finding 3: Phase 2 "verified facts" claim about Postgres auto-scaling is empirically false

- **Severity:** Medium
- **Location:** Phase 2, section "Context / verified facts"
- **Flaw:** *"Postgres runs stock `postgres:16-alpine`, no `command:`/`postgresql.conf` override → auto-scales `shared_buffers`/`max_connections` off container-visible memory. No explicit high setting to preserve."* (`phase-02-resource-limits.md:13`). This is presented under a "verified facts" heading but is not true of the stock image.
- **Failure scenario:** Empirically tested: `docker run -d --memory=1g postgres:16-alpine` (no config override) → `SHOW shared_buffers` = `128MB`, `SHOW max_connections` = `100`, regardless of the container's memory limit. The stock image does **not** read cgroup memory and auto-tune `postgresql.conf` — it ships fixed `initdb` defaults. In this specific case the wrong premise happens to be *safe* (real footprint is smaller than the plan assumed, so the 1g ceiling has more margin than the "auto-scales" framing implies), but it means the "no explicit high setting to preserve" reasoning is coincidentally, not verifiably, correct. If a future tuning pass assumes Postgres will "auto-scale up" to use the full 1g ceiling for cache, it won't — `shared_buffers` will stay at 128MB and most of the 1g ceiling is headroom, not usable cache, unless someone adds an explicit `command:`/`postgresql.conf` override.
- **Evidence:** Direct empirical test (this review): `docker run -d --name pgtest -e POSTGRES_PASSWORD=x --memory=1g postgres:16-alpine` → `psql -c "SHOW shared_buffers; SHOW max_connections;"` → `128MB` / `100`. Plan claim: `phase-02-resource-limits.md:13`.
- **Suggested fix:** Correct the claim to: "stock image uses fixed low defaults (shared_buffers 128MB, max_connections 100), independent of the container memory limit — the 1g ceiling is a safety cap, not a tuning target. If DB performance under load requires more cache, an explicit `command:`/`postgresql.conf` override is needed and is currently absent."

## Finding 4: `deploy.resources` mechanism is correctly claimed to work on non-swarm `docker compose`, but the plan never verifies the VPS's actual Compose version supports it

- **Severity:** Medium
- **Location:** Phase 2, section "Compose v2 reservation caveat"
- **Flaw:** The plan asserts *"`docker compose up` (v2, non-swarm) reliably applies `limits` (`--memory`, `--cpus`)"* (`phase-02-resource-limits.md:49`). **VERIFIED TRUE** by direct empirical test in this review (Compose CLI v5.1.4: `deploy.resources.limits.memory: 128m` → container `HostConfig.Memory = 134217728`; `limits.cpus: '0.5'` → `HostConfig.NanoCpus = 500000000`; `reservations.memory: 64m` → `HostConfig.MemoryReservation = 67108864` applied; `reservations.cpus` → `HostConfig.CpuShares = 0`, i.e. genuinely a no-op, confirming the plan's own caveat about reservations.cpus). So the *mechanism* claim is correct for a modern Compose v2 CLI. However, the plan's only cited prod evidence that this pattern already works is *"Only Jenkins (`docker-compose.jenkins.yml:36-40`, 1.5 CPU / 3 GiB) does [have `deploy.resources`]"* — it treats this as proof the pattern is live and working, but never runs `docker inspect cmcnew-jenkins --format '{{.HostConfig.Memory}}'` (or equivalent) against the **actual running VPS container** to confirm the VPS's Compose version actually enforces it there. `docs/prod-deploy-security-runbook.md:12` only says *"Docker + docker compose v2"* — no version pin.
- **Failure scenario:** If the VPS runs an older Compose v2 release predating non-swarm `deploy.resources` support (this support was not present in the earliest v2 CLI releases), all 8 new `deploy.resources` blocks silently no-op on the VPS even though `docker compose -f ... config` parses cleanly (a YAML-valid but functionally inert config) — the plan's own listed validation step (`phase-02-resource-limits.md:59`, "config parses") would pass while providing zero actual protection, and the Phase 4 soak (which does watch `docker stats`) is the *only* thing that would catch this — a 48h soak is a very late/expensive way to discover a version-support gap.
- **Evidence:** Empirical test (this review, Compose v5.1.4). `phase-02-resource-limits.md:49`; `docs/prod-deploy-security-runbook.md:12`; `docker-compose.jenkins.yml:36-40`.
- **Suggested fix:** Add an explicit Phase 4 Step 0 pre-flight: `docker inspect cmcnew-jenkins --format '{{.HostConfig.Memory}} {{.HostConfig.NanoCpus}}'` on the VPS to confirm the *existing* Jenkins limit is actually enforced (not just declared) before trusting the same mechanism for 8 more services.

## Finding 5: Plan is silent on the existing `.github/workflows/ci.yml`, which already posts a competing, permanently-failing check on every PR

- **Severity:** High
- **Location:** Phase 3, entire phase (no mention of `.github/workflows/`)
- **Flaw:** `manhquydev/CMCnew` already has `.github/workflows/ci.yml` (VERIFIED, read in full) — a workflow that runs on every `pull_request` and `push: [main]`, spins up Postgres, does install/migrate/seed/typecheck/test/build, and per `AGENTS.md` ("`ship`/`review-pr`/`vibe --ship`/`team` stay deferred until a green CI exists (GitHub Actions billing is blocked → runs fail at ~3s...)") **always fails within ~3 seconds** due to a billing block. Phase 3 designs a brand-new `CMCnew CI` context sourced from Jenkins and a `gh api` branch-protection call that only requires `contexts: ["CMCnew CI"]` — it never addresses the pre-existing, always-red `CI / build` GitHub Actions status that will keep posting on every PR alongside it.
- **Failure scenario:** Two named checks now appear on every PR: `CMCnew CI` (the new, real gate) and `CI / build` (GitHub Actions, permanently failing, not required, therefore cosmetic noise). This is confusing for the solo maintainer ("is my PR blocked by the GH Actions X or not?") and is a duplicated/parallel CI pipeline that does *the same work Jenkins now also does* (install, migrate, typecheck, test, build) on every push — silently burning GitHub Actions minutes/quota for a job whose result is never used and always fails. This is exactly the kind of "parallel reimplementation of existing pipeline" scope gap the plan should flag but doesn't; none of the three items or Phase 4 rollback/cleanup steps mention disabling or removing `.github/workflows/ci.yml`.
- **Evidence:** `.github/workflows/ci.yml:1-74` (full read, confirmed `on: pull_request` + `push: [main]`, job name `build`, does install/migrate/seed/typecheck/test/build — the same coverage Jenkins's `Lint + Typecheck` + `Integration tests` stages now provide); `AGENTS.md` ("GitHub Actions billing is blocked → runs fail at ~3s"). Phase 3 file scope (`phase-03-ci-merge-gate.md:60-64`, "Files") lists only `Jenkinsfile` and `scripts/setup-github-required-check.sh` — no `.github/workflows/` file is touched or even mentioned anywhere in the plan.
- **Suggested fix:** Either explicitly scope-exclude `.github/workflows/ci.yml` with a stated reason (e.g. "keep as a future fallback if Actions billing is unblocked, ignore for now"), or fold "disable/delete the redundant Actions workflow" into Phase 3/4 so the plan's own PR checklist doesn't leave two conflicting CI signals on every PR going forward.

## Finding 6: Cloudflare "Full" mode premise is correct, but the plan asserts it without checking the account's actual configured SSL/TLS mode

- **Severity:** Medium
- **Location:** Phase 1, section "Decision: canonical = self-signed origin cert behind Cloudflare 'Full'"
- **Flaw:** The claim *"Cloudflare already owns the browser-trusted edge cert. The origin cert only secures the Cloudflare↔origin hop; it does not need public trust"* (`phase-01-tls-reconciliation.md:21`) is **VERIFIED CORRECT** as a general Cloudflare SSL/TLS-mode fact: "Full" mode encrypts the edge↔origin hop without validating the origin cert's CA/issuer (only "Full (strict)" validates against a trusted CA), per Cloudflare's own SSL/TLS mode docs. The plan's premise about the *protocol* is sound. What is **not verified anywhere in the plan** is that the live Cloudflare zone for `erp.cmcvn.edu.vn`/`hoc.cmcvn.edu.vn` is actually set to "Full" (vs. "Full (strict)", which the memory index and this plan both imply but never confirm by checking the Cloudflare dashboard/API). Phase 1's context section says only "Project memory: prod is Cloudflare-proxied" — memory of proxying, not of the specific SSL mode.
- **Failure scenario:** If the zone is actually already set to "Full (strict)" (e.g. changed by a prior manual dashboard action, undocumented in this repo), a self-signed cert would be **rejected by Cloudflare's edge**, producing Error 526 (invalid origin certificate) for all traffic — a full site outage — the first time `ensure-origin-cert.sh` regenerates a self-signed cert on an empty volume in production, and the fail-safe design in Phase 1/4 ("verify only presence + parseability + SAN, not issuer identity") would not catch this because the failure happens at Cloudflare's edge, not inside the deploy.
- **Evidence:** Cloudflare docs (web search, this review): "Full (strict) ... added validation of the origin server's certificate ... must be ... issued by a publicly trusted CA or Cloudflare's Origin CA." Plan text: `phase-01-tls-reconciliation.md:14-24` (no Cloudflare API/dashboard check cited anywhere in the plan).
- **Suggested fix:** Add a one-line pre-flight to Phase 4 Step 0 or Step 3: confirm the live Cloudflare zone SSL/TLS mode via `curl -s https://api.cloudflare.com/client/v4/zones/{id}/settings/ssl` (or dashboard screenshot) is exactly "Full" (not "Full (strict)") before the first self-signed-cert regen touches the live volume.

## Finding 7: `gh api` branch-protection `contexts` claim is directionally fine, but the plan doesn't verify PAT `checks:write` actually differs from the SCM scope already in use

- **Severity:** Medium
- **Location:** Phase 3, section "GitHub branch protection — scripted required check" + "Operator pre-flight"
- **Flaw:** The plan correctly identifies (`phase-03-ci-merge-gate.md:70` unresolved question #1 in `plan.md:74`) that PAT scope is unverifiable from the repo, and gates on it. This is good practice, not a flaw — but the plan's suggested verification command, `gh api -H "Authorization: token $PAT" /repos/manhquydev/CMCnew/check-runs`, is not a valid GitHub REST endpoint (check-runs is scoped under a specific commit SHA: `/repos/{owner}/{repo}/commits/{ref}/check-runs`, or creation requires `POST /repos/{owner}/{repo}/check-runs`) — as written it will 404, which an operator running it verbatim would likely misread as a scope failure (403) rather than a wrong-endpoint (404), muddying the exact pre-flight gate the plan itself calls "critical" ordering.
- **Failure scenario:** Operator runs the exact command from the plan, gets 404, incorrectly concludes the PAT lacks `checks:write`, spends time regenerating a PAT that was already fine — wasted effort but not a prod risk. Low material impact, included because the plan explicitly asks operators to run this as a go/no-go gate before merging Phase 3.
- **Evidence:** `phase-03-ci-merge-gate.md:35` (`gh api -H "Authorization: token $PAT" /repos/manhquydev/CMCnew/check-runs` — not a valid unscoped GH REST path).
- **Suggested fix:** Use a valid pre-flight, e.g. `gh api -H "Authorization: token $PAT" repos/manhquydev/CMCnew/commits/main/check-runs` (read-scope check) or attempt an actual `POST .../check-runs` against a disposable ref to positively confirm write access, since a GET succeeding doesn't prove `checks:write` either.
