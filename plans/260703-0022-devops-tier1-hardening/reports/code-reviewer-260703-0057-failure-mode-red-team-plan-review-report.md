# Red-Team Plan Review — DevOps Tier-1 Hardening (Failure-Mode Analyst)

Scope: `plans/260703-0022-devops-tier1-hardening/{plan.md,phase-01..04}.md`. This is a plan-quality review with codebase fact-checking (no code changes made). Verification method: grep/glob against the live repo tree, not the plan's own claims.

## Finding 1: Certbot renewal disabled by default on an unverified assumption about which cert type is actually live
- **Severity:** Critical
- **Location:** Phase 1, section "4. `docker-compose.prod.tls.yml` — dormant certbot"; Phase 4, Step 3
- **Flaw:** Phase 1 asserts "The `certbot renew` loop is meaningless for a self-signed 10-year cert" and gates `certbot` behind `profiles: ['le']` (off by default). This assumes the VPS's live `cmcnew-prod_letsencrypt` volume already holds a self-signed cert. Nothing in Phase 1 or Phase 4 adds a step to actually read the live cert's issuer/type on the VPS before landing this change. Phase 4 Step 3.3 only says the helper "sees the cert present, verifies, proceeds" — but the helper's verify (Phase 1 §1.3) checks only parseability + SAN coverage + `checkend 0`, which passes identically for a self-signed OR a real Let's Encrypt cert. Verification is (correctly) strategy-agnostic, but that same agnosticism means the plan never distinguishes the two cases to decide whether disabling `certbot renew` is safe.
- **Failure scenario:** If the live volume actually contains a real 90-day LE cert (obtained via `scripts/prod-tls-bootstrap.sh`, which the compose file's own header still instructs as canonical step 1 — see evidence below), gating `certbot` behind an opt-in profile silently stops the 12h `certbot renew` loop. The existing valid cert keeps passing `ensure-origin-cert.sh`'s checks (not expired yet, correct SANs) for up to ~90 days, then expires with no automated renewal running. nginx keeps serving the now-expired cert; Cloudflare "Full" mode's origin TLS handshake starts failing, causing a full outage for both `erp.cmcvn.edu.vn` and `hoc.cmcvn.edu.vn` with no deploy-time signal (the helper only checks presence/parse/SAN at deploy time, which was months earlier).
- **Evidence:**
  - `docker/docker-compose.prod.tls.yml:6-13` (current header, unmodified until this plan lands): `# 1. one-time TLS cert (port 80 free): bash scripts/prod-tls-bootstrap.sh` — the file's own currently-documented canonical bootstrap step is the **Let's Encrypt** path, not self-signed. This directly contradicts Phase 1's premise that self-signed is what's live.
  - `scripts/prod-tls-bootstrap.sh:20-24` — real certbot `certonly --standalone -d erp.cmcvn.edu.vn -d hoc.cmcvn.edu.vn`, same SAN coverage as the self-signed path, so `ensure-origin-cert.sh`'s verify step (Phase 1 §1.3) cannot distinguish which was used.
  - Phase 1 plan text itself (line 24) acknowledges this exact ambiguity is unresolved: "Trade-off accepted: origin traffic encrypted but not authenticated" — but never resolves *which* cert is currently live before deciding to disable renewal for it.
  - Phase 4 has no step reading `openssl x509 -in fullchain.pem -noout -issuer` (or similar) against the real `cmcnew-prod_letsencrypt` volume before merging the certbot-profile-gating change.
- **Suggested fix:** Add a mandatory Phase 4 Step 0 (or early Step 3) sub-step: SSH to the VPS and run `docker run --rm -v cmcnew-prod_letsencrypt:/le alpine sh -c "apk add --no-cache openssl >/dev/null; openssl x509 -in /le/live/erp.cmcvn.edu.vn/fullchain.pem -noout -issuer -enddate"`. If issuer is `Let's Encrypt`, either (a) do not gate certbot behind `profiles: ['le']` until the cert is deliberately rotated to self-signed, or (b) run `prod-tls-bootstrap.sh`'s self-signed equivalent to replace it first. Do not assume from source-reading alone.

## Finding 2: The docker build phase preceding `api-migrate` is unbounded and excluded from both the memory budget and the soak validation
- **Severity:** Critical
- **Location:** Phase 2, "Sizing arithmetic" and "Proposed limits" table (`api-migrate` row); Phase 4, Step 4
- **Flaw:** Phase 2 assigns `api-migrate` a 768m ceiling ("has headroom; prisma migrate is moderate") and treats it as excluded from the "steady-state" sum since it's one-shot. But `deploy.resources.limits` only bounds the **runtime container**, not the `docker build` invocation that produces its image. Per the Jenkinsfile, the build for `api-migrate`'s image (`apps/api/Dockerfile`: full monorepo `pnpm install --frozen-lockfile`, `prisma generate`, TypeScript source copy) happens as part of `$COMPOSE --profile migrate run --rm api-migrate` (compose builds on-demand if the image is stale/missing) — and this occurs **before** `$COMPOSE up -d --build` in the deploy sequence, i.e., concurrently with `postgres`+`redis` already running (1g+256m reserved) and Jenkins's own worst-case 3 GiB.
- **Failure scenario:** `pnpm install --frozen-lockfile` across the full workspace plus `prisma generate` is the single heaviest, least-bounded operation in the entire deploy path, yet it is invisible to Phase 2's "steady-state ≤ 2.8 GiB" arithmetic and to Phase 4's 48h `docker stats`/OOM soak (which only samples long-running services — the build process exits before/during container creation and isn't a `deploy.resources`-bound entity to sample). A build-phase memory spike stacked on top of Jenkins (up to 3 GiB) + postgres/redis (1.25 GiB) on a 2 vCPU / 7.8 GiB no-swap box is a plausible full-host OOM that kills postgres (the highest-priority-to-protect service per Phase 2's own reservation logic) via the kernel OOMKiller, not the cgroup limit — which the plan's rollback/soak strategy has no visibility into because it wasn't modeled.
- **Evidence:** `docker/docker-compose.prod.tls.yml:99-107` (`api-migrate` has `build: {context: .., dockerfile: apps/api/Dockerfile}` and `profiles: ['migrate']`); `Jenkinsfile:47-49` runs `$COMPOSE --profile migrate run --rm api-migrate` **before** `$COMPOSE up -d --build` on line 50 — so the build for at least the api image is forced to happen at the `run` step, not deferred to the later `--build` flag. Phase 2's own Risks table (line 68) only flags "CI build's unbounded `node:22-alpine` sibling container (Jenkinsfile:26)" — a *different* container (the Lint+Typecheck stage's ephemeral `docker run`), not the compose `build:` sections used by `api`/`api-migrate`/`api-seed`/`admin`/`lms`, which the plan never mentions at all.
- **Suggested fix:** Extend Phase 2's Files/Risks section to explicitly note that `deploy.resources` does not bound `docker compose ... build` or the implicit build-on-run behavior, and add a Phase 4 validation step that samples host-level (not container-level) memory during the actual `run --rm api-migrate` + `up -d --build` window on a real deploy, not just the post-deploy steady-state soak.

## Finding 3: `catchError`-wrapped `publishChecks` can leave GitHub silently blocking all merges while Jenkins shows green/unstable, with no detection path
- **Severity:** High
- **Location:** Phase 3, section "1. Jenkinsfile — publish one check"; Phase 4, Step 2
- **Flaw:** Phase 3 deliberately wraps every `publishChecks` call in `catchError(buildResult: null, stageResult: 'UNSTABLE')` so "a checks-API hiccup never fails the pipeline." Once Phase 4 Step 2 enables branch protection requiring the `CMCnew CI` context, this design means: if the checks API call fails for any reason *after* the required check is live (Jenkins restarted and `github-checks` plugin didn't reload, PAT revoked/expired, GitHub outage, controller CASC drift), the Jenkins pipeline itself keeps reporting green/unstable (misleading — "the check is a reporter, not a gate inside Jenkins" per the plan's own words), while GitHub shows the `CMCnew CI` context as perpetually pending/never-reported, which blocks **every** PR merge on the repo. There is no monitoring, alert, or even a documented smoke-check for "is the check still posting" — only a manual `gh api DELETE` escape hatch that nobody is prompted to run because the failure is invisible in the system operators actually watch (Jenkins UI).
- **Failure scenario:** Weeks after Phase 4 lands, the operator does an unrelated Jenkins controller restart/rebuild (e.g. routine `docker compose -f docker/docker-compose.jenkins.yml up -d --build` for an unrelated plugin update) that doesn't cleanly reload `github-checks`, or the `github-token` PAT expires (GitHub PATs can have expiry dates). `publishChecks` starts silently failing via `catchError`; Jenkins pipelines keep going green. All subsequent PRs sit permanently un-mergeable at GitHub with no Jenkins-side signal that anything is wrong — a single-maintainer repo (per `AGENTS.md` branch workflow) could go a long time before someone opens a PR, sees it stuck, and has to reverse-engineer that the *check-publishing*, not the *code*, is broken.
- **Evidence:** Phase 3 lines 22-24 (design of `catchError`), Phase 3 Risks table row "PAT lacks `checks:write` → checks silently 403 | ... | catchError keeps pipeline green so a scope gap doesn't block deploys" — the plan's own text confirms the silent-green behavior is intentional and treats it as a feature for the deploy path, without separately addressing that the *same* silence is a liability once branch protection depends on that check posting.
- **Suggested fix:** Add an explicit Phase 4 post-rollout monitoring step: either (a) a scheduled Jenkins job or cron that posts a canary check weekly and alerts if it fails, or (b) at minimum, document in the runbook a manual "first sign of a stuck PR → check `gh api repos/manhquydev/CMCnew/commits/<sha>/check-runs` before assuming it's a code problem" troubleshooting note, since none exists today.

## Finding 4: `docker compose run --rm` resource-limit enforcement for one-shot jobs is asserted, not verified — and Phase 4's soak structurally cannot catch a violation
- **Severity:** High
- **Location:** Phase 2, "Compose v2 reservation caveat" section; Phase 4, Step 4 ("Soak ≥ 48h")
- **Flaw:** Phase 2 documents (correctly, per Compose behavior) that `reservations.cpus` may be a no-op outside Swarm, but asserts `limits` (`--memory`, `--cpus`) are "reliably applied" without distinguishing `docker compose up` (used for `postgres`/`redis`/`api`/`admin`/`lms`/`nginx`) from `docker compose run --rm` (used for `api-migrate`/`api-seed` per `Jenkinsfile:47`). Phase 4's validation matrix and 48h soak explicitly target only the long-running services ("Watch postgres... and api... especially") — `api-migrate`/`api-seed` run once at deploy time and exit before the soak window even meaningfully begins, so the plan's own "Done =" criterion ("numbers tuned-and-locked... not guessed-and-shipped") is never actually satisfied for these two services; their 768m ceilings ship un-soaked and un-observed on every future deploy indefinitely.
- **Failure scenario:** If `docker compose run` on this Compose version (`v5.1.4` confirmed installed) applies `deploy.resources.limits` differently than `up` (e.g. only `--cpus` translated, not `--memory`, or vice versa — behavior has changed across Compose v2 minor versions historically), the 768m ceiling silently does not bind `api-migrate`, and a future migration (schema now has 69 files and growing) that touches a larger table than today's could consume more memory than budgeted with no cgroup backstop, right at the point in the deploy where postgres is also running unguarded by anything but its own 1g limit — the exact concurrency window Finding 2 also flags.
- **Evidence:** `Jenkinsfile:47` — `$COMPOSE --profile migrate run --rm api-migrate` (uses `run`, not `up`). Phase 2 file (`phase-02-resource-limits.md:59-61`) validation section only lists `compose config` parsing + the 48h soak of steady-state services — no `run --rm` specific limit-enforcement check (e.g. `docker inspect` the transient container's `HostConfig.Memory` during a migrate run) is proposed anywhere in Phases 2 or 4.
- **Suggested fix:** Add to Phase 4 Step 3/4: during the first real `main` deploy after Phase 2 lands, capture `docker inspect <api-migrate container id> --format '{{.HostConfig.Memory}}'` (or watch it briefly via `docker stats` in a second terminal) to confirm the limit is actually attached to the one-shot container, not just parsed successfully by `compose config`.

## Finding 5: Postgres auto-tuning claim is factually incorrect, undermining the sizing rationale (numbers happen to still be safe, but for the wrong stated reason)
- **Severity:** Medium
- **Location:** Phase 2, "Context / verified facts" (line 13)
- **Flaw:** The plan states: "Postgres runs stock `postgres:16-alpine`, no `command:`/`postgresql.conf` override → auto-scales `shared_buffers`/`max_connections` off container-visible memory." This is false — the stock `postgres` Docker image does **not** auto-tune `shared_buffers` or `max_connections` based on cgroup-visible memory; those remain at Postgres's compiled-in defaults (`shared_buffers=128MB`, `max_connections=100`) unless explicitly set via `command:`, a mounted `postgresql.conf`, or an extension like pgtune. Confirmed via `docker-compose.prod.tls.yml:20-32` — no `command:` override, no config mount for `postgres`.
- **Failure scenario:** Not a live-outage risk by itself (the false premise happens to be more conservative than reality would require, so the 1g ceiling is not endangered by it) — but it means the "No explicit high setting to preserve" reasoning used to justify the 1g/512m postgres numbers is built on a wrong mental model of how the container behaves under load. If a future engineer reads this doc and adds a `POSTGRES_SHARED_BUFFERS`-style override believing the memory footprint "auto-scales" already, they could double-allocate.
- **Evidence:** `docker/docker-compose.prod.tls.yml:20-32` — no `command:`, no `postgresql.conf` volume mount for the `postgres` service.
- **Suggested fix:** Correct the claim to: "stock config, defaults (`shared_buffers=128MB`, `max_connections=100`) apply regardless of container memory limit — no auto-tuning." Numbers can stay as-is; only the stated rationale needs correcting.

## Finding 6: CI-triggered ephemeral containers are only partially flagged, and the flagged instance understates the actual footprint
- **Severity:** Medium
- **Location:** Phase 2, Risks table (line 68); Phase 4 ("Why order matters")
- **Flaw:** Phase 2's risk table flags exactly one unbounded CI container: "CI build's unbounded `node:22-alpine` sibling container (Jenkinsfile:26)." In reality there are at least three unbounded, unaccounted containers that can run during CI on the same 2 vCPU/7.8 GiB box: (1) `Jenkinsfile:26` node:22-alpine for Lint+Typecheck, (2) a second `node:22-alpine` inside `scripts/ci-integration-tests.sh` that itself runs `pnpm install --frozen-lockfile` + `prisma generate` + `prisma migrate`, and (3) a full ephemeral `postgres:16-alpine` container bound to host port `55432` for the integration-test run. None of these three are bound by `deploy.resources` (they're plain `docker run`, not part of the compose stack Phase 2 modifies), and `Jenkinsfile`'s `disableConcurrentBuilds()` only prevents concurrent runs of the *same* branch job — it does not prevent a `changeRequest()` (PR) build's Lint/Integration stages from running concurrently with a `main` branch's Build+Deploy stage in Jenkins multibranch pipelines, since each branch/PR is a distinct job.
- **Failure scenario:** A PR opened at the same time as (or during) a `main` deploy spins up a second full ephemeral Postgres + two more Node containers, none capped, concurrently with the just-added `deploy.resources`-bounded prod stack and Jenkins's own 3 GiB ceiling — compounding exactly the contention risk Finding 2 describes, but the plan's own Risks table undercounts it (mentions 1 of 3 unbounded containers) and defers it as "FLAG for a follow-up; not fixed in this phase" without acknowledging the other two.
- **Evidence:** `scripts/ci-integration-tests.sh:10-28` — `docker run -d --rm ... -p 55432:5432 postgres:16-alpine` plus two separate `docker run --rm ... node:22-alpine` invocations (`pnpm install` + `prisma generate`/`migrate`, then the test run itself), none with `--memory`/`--cpus` flags. `Jenkinsfile:37` — Integration tests stage runs `when { anyOf { branch 'main'; changeRequest() } }`, i.e., on every PR, and `disableConcurrentBuilds()` (`Jenkinsfile:9`) is a per-job (per-branch) option in declarative pipelines, not a cross-job/global lock.
- **Suggested fix:** Either broaden the existing "FLAG for follow-up" note to list all three unbounded CI containers (accurate risk sizing even if deferred), or add `--memory`/`--cpus` flags to the `docker run` invocations in `ci-integration-tests.sh` and the Lint stage as a cheap inclusion in this same plan, since the fix is a one-line flag addition per invocation, not new scope.
